The Anomaly Engine only runs on requests that passed all rule checks. Detection layers are independent — each flagged event is attributed to exactly one source. It answers: "This request looks valid on inspection, but is this client behaving normally?"
Why rules alone are insufficient — click each gap to expand
►
The threshold gap — what a rate limiter can't see
The rate limiter blocks IPs exceeding 200 req/10s. But an attacker sending 180 req/10s is invisible to it. Or consider a client that normally sends 5 req/s suddenly jumping to 50 req/s — still under the global limit, but a 10× change from their personal baseline is suspicious. The rule engine has no memory of past behaviour. The anomaly engine does.
►
Per-IP baseline profiling — each client gets their own threshold
For each IP, a sliding window (last 30 seconds) tracks inter-arrival times (IAT) and payload sizes. After 150 samples, every new request is scored against this IP's own baseline — not a global threshold. A heavy API user and a light user have different baselines. An attacker impersonating a heavy user still gets flagged if their pattern suddenly changes.
►
Slow-rate limitation — what the anomaly engine also misses
If an attacker sends requests slowly enough and consistently enough, the baseline adapts to include them. Measured in experiments: during a slow-rate attack, the baseline median collapsed from ~100ms IAT to 4.34ms within one 30s window cycle. Subsequent bursts scored Z ≈ 0.97, well below the threshold of 1.5. Fix: freeze the baseline after warmup, or use a longer (≥120s) window.
Modified Z-Score — click any highlighted term for its meaning
# Modified Z-score (Iglewicz & Hoaglin, 1993)
Z = 0.6745 × |x − median(baseline) | / MAD(baseline)

# Where MAD = Median Absolute Deviation
MAD = median( |xi − median(X) | )

# Flag condition
Z > 1.5 AND x < median (IAT acceleration) → flag
Why modified Z-score? Standard Z uses mean and std dev, which are shifted by outliers. A single burst would widen the std dev, making the next burst look less anomalous. Median and MAD are robust to outliers — the baseline stays stable even when attack traffic mixes in.
IAT distribution: baseline vs. burst
Baseline IAT (~100ms) Burst IAT (~1ms) — Z > 1.5
►
Directional scoring
Only faster-than-baseline is flagged
A long pause between requests (high IAT) is not an attack — it might be the client thinking or rate-limiting itself. Only when IAT is shorter than the baseline median is the request scored. The condition is x < median(baseline).
►
IAT-only weighting
Payload Z-score computed but weighted 0%
Different task types have naturally different payload sizes. Scoring payload size would generate false positives whenever a client changes task type. IAT, however, is task-type-independent — it measures request pace, not content.
►
150-sample warmup
No scoring until baseline is stable
Computing a Z-score from 2 samples is meaningless. 150 samples at ~10 req/s takes ~15 seconds. Flood IPs (blocked at 200 req/10s) never accumulate 150 samples before being blocked, so the warmup guard is effectively only relevant for slow-rate anomaly detection.
►
30-second sliding window
Baseline adapts to legitimate rate changes
If a legitimate client doubles their request rate permanently, the baseline adapts within 30 seconds and stops flagging them. This prevents false positives for natural usage changes. The tradeoff: sustained slow-rate attacks can contaminate the baseline within the same window.
