The Anomaly Engine only runs on requests that passed all rule checks. This keeps detection layers independent — each flagged event is attributed to exactly one source. It answers the question: "This request looks valid, but is the client behaving normally?"
Why rules alone aren't enough
The gap in rule-based detection
The rate limiter blocks IPs exceeding 200 req/10s. But what about an attacker sending 180 req/10s? That's under the threshold — the rule engine sees nothing wrong.

Or consider a legitimate client that normally sends 5 req/s suddenly jumping to 50 req/s. Still under the rate limit, but a 10x change from their personal baseline is suspicious.
Per-IP baseline profiling
For each IP, we maintain a sliding window (last 30 seconds) of:
  • Inter-arrival times (IAT) — gap between consecutive requests
  • Payload sizes — raw body bytes per request
After 150 samples, each new request is scored against this IP's own baseline. A client with stable 100ms IAT suddenly sending at 1ms IAT gets flagged — even though the global rate limit hasn't been reached.
Modified Z-Score (Iglewicz & Hoaglin, 1993)
# Modified Z-score formula
Z = 0.6745 × |x - median(baseline)| / MAD(baseline)

# Where MAD = Median Absolute Deviation
MAD = median(|xi - median(X)|)

# 0.6745 normalises MAD to match standard deviation under normality
# Threshold: Z > 1.5 → flagged as anomalous
Why modified Z-score instead of standard Z-score?
Standard Z-score uses mean and standard deviation, which are heavily influenced by outliers. A single burst request would shift the mean and inflate the std dev, making subsequent burst requests look less anomalous. The modified version uses median and MAD, which are robust to outliers — the baseline stays stable even when attack traffic is mixed in.

Key design decisions
Directional scoring Only flags faster-than-baseline IAT. A pause (long IAT) is not an attack.
IAT-only weighting Payload Z-score computed but weighted 0% — task types have naturally different sizes.
150-sample warmup No scoring until baseline is stable. Flood IPs (blocked at 200) never warm up.
30s sliding window Baseline adapts to legitimate rate changes within 30 seconds.
