Key findings
Finding 5 — Slow-Rate Evasion
Baseline Contamination: Why Slow-Rate Evades Detection
The anomaly engine fails against sustained micro-burst attacks because the 30-second rolling baseline is contaminated by the attack traffic itself. Early burst requests collapse the baseline median from ~100 ms to 4.34 ms within one window cycle. All subsequent burst requests then score Z ≈ 0.97 — well below threshold = 1.5.

This is not a threshold calibration issue. It is an architectural property of short-window baselines under sustained attacks. A frozen baseline (computed during warmup only, not updated during scoring) or a ≥120 s window would prevent contamination.
# Z-score measurements during slow-rate attack phase (200 requests) IAT Z-score mean 0.304 IAT Z-score max 1.171 (threshold = 1.5) Requests above threshold 0 Baseline IAT median 4.34 ms (was ~100 ms at warmup) Baseline IAT MAD 2.32 ms # Z-score calculation (verified) Z = 0.6745 × |1ms − 4.34ms| / 2.32ms ≈ 0.97 ← max observed: 1.171
Fix: Freeze the baseline after warmup, or use a ≥120 s window. Warmup samples remain in the window throughout the attack and keep the median at ~100 ms.
Finding 7 — Routing × Security
Source Hashing Amplifies Single-Source Attack Impact
Source hashing routes all requests from the same IP to the same backend server. Under a single-source ramp attack, the targeted server becomes overloaded before IDMS escalation fires — producing backend 503 errors independently of whether detection occurs.

Round-robin and least-loaded distribute attack traffic across all 4 servers, preventing premature saturation. This demonstrates that load balancing algorithm choice affects the blast radius of a single-source attack independently of the IDMS.

Implication: Security and routing interact non-trivially. An inline IDMS does not eliminate the need to consider algorithm choice when defending against single-source attacks.

Honest evaluation
Limitations
Limitation 1
Single-Node IDMS State
All per-IP state (rate windows, anomaly baselines, SQLite blocks) is local to one instance. Horizontal scaling requires centralised state (Redis) or consistent IP-to-instance routing. Standard limitation of inline inspection proxies.
Limitation 2
Slow-Rate Detection Requires Environment Calibration
Threshold = 1.5 is appropriate for low-jitter environments. Docker Desktop on Windows adds 20–80 ms jitter that widens IAT baselines and suppresses Z-scores. A frozen baseline (warmup-only, not updated during scoring) or a ≥120 s window would prevent contamination.
Limitation 3
Rate-Limit Ceiling Is Structural
The first 200 requests in any 10-second window always pass before the block fires. This is unavoidable with a sliding-window rate limiter. Mitigation: lower the threshold, or add a per-connection rate limit.
Limitation 4
Controlled Evaluation Environment
4 backend servers, simulated traffic patterns, single attacker IP per mode. Real-world traffic is more diverse. Findings characterise system behaviour under controlled conditions, not production traffic patterns.
