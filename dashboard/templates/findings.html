Finding 5 Baseline Contamination — Why Slow-Rate Evades Detection ►
The anomaly engine fails against sustained micro-burst attacks because the 30-second rolling baseline is contaminated by the attack traffic itself. Early bursts collapse the baseline median from ~100ms to 4.34ms within one window cycle, making subsequent bursts score Z ≈ 0.97 — below threshold = 1.5. This is architectural, not a calibration problem. A frozen baseline or ≥120s window would prevent it.
# Measured Z-scores during slow-rate attack (200 requests) IAT Z-score mean 0.304 IAT Z-score max 1.171 (threshold = 1.5) Requests above threshold 0 Baseline median at scoring4.34 ms (was ~100 ms at warmup) Baseline MAD at scoring 2.32 ms # Verification Z = 0.6745 × |1ms − 4.34ms| / 2.32ms ≈ 0.97
Fix: Freeze the baseline after warmup. Warmup samples remain in the window throughout the attack, keeping the median at ~100ms. The same burst then scores Z ≈ 62.
Finding 7 Source Hashing Amplifies Single-Source Attack Impact ►
Source hashing routes all requests from the same IP to the same backend server. Under a single-source ramp attack, the targeted server saturates before IDMS escalation fires, producing 503 errors independent of detection. Round-robin and least-loaded distribute traffic across all 4 servers, preventing premature saturation. Algorithm choice affects blast radius independently of IDMS.
# SH ramp: backend 503s from a single targeted server Step 3 (11 req/s) 41 backend 503s Step 4 (14 req/s) 72 backend 503s RR / LL 503s 0 at equivalent steps # Step 5 detection rates (3-run averages) Round Robin 51.5% ← distributed, no saturation Source Hashing 46.3% ← server saturates before block fires Least Loaded 52.5% ← distributed, no saturation
Implication: An inline IDMS does not eliminate the need to consider algorithm choice when defending against single-source attacks. Security and routing interact non-trivially.

Limitations — click to expand
►
Single-Node IDMS State
All per-IP state (rate windows, anomaly baselines, SQLite blocks) is local to one instance. Horizontal scaling requires centralised state (Redis) or consistent IP-to-instance routing. Standard limitation of inline inspection proxies.
►
Slow-Rate Detection Requires Calibration
Threshold=1.5 is appropriate for low-jitter environments. Docker Desktop on Windows adds 20–80ms jitter that widens IAT baselines and suppresses Z-scores. A frozen baseline (warmup-only, not updated during scoring) or a ≥120s window would prevent contamination.
►
Rate-Limit Ceiling Is Structural
The first 200 requests in any 10-second window always pass before the block fires. This is unavoidable with a sliding-window rate limiter. Mitigation: lower the threshold, or add a per-connection rate limit before the sliding window.
►
Controlled Evaluation Environment
4 backend servers, simulated traffic patterns, single attacker IP per mode. Real-world traffic is more diverse. Findings characterise system behaviour under controlled conditions, not production traffic patterns.
