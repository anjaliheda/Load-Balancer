Algorithms — click to switch
Round Robin
Cycles through servers in fixed order: 1 → 2 → 3 → 4 → 1 …
Simple, predictable, and ensures even distribution when all servers are equal.
Our improvements
  • Background health checks — unhealthy servers are skipped automatically (5s polling interval instead of per-request HTTP calls)
  • Thread-safe iterator — concurrent requests can't skip or double-assign servers
  • Exponential backoff — failed servers get increasing cooldown before re-check
Source Hashing
Hashes the request content (task type + parameters) to pick a server.
Identical requests always go to the same server — great for caching.
Our improvements
  • Content-aware hashing — hash key includes task type, parameters, and time bucket for balanced distribution
  • Fallback on failure — if the hashed server is down, tries the next server in ring order
  • Optimized hash — uses first 8 hex chars of MD5 instead of full 32 for faster computation
Least Loaded
Queries each server's current load and picks the one with the fewest active requests.
Best under uneven workloads where tasks have different processing times.
Our improvements
  • Real-time load queries — polls each server's /load endpoint for live metrics
  • Jitter tie-breaking — adds small random noise to prevent all requests hitting the same "lightest" server simultaneously
  • Guaranteed load decrement — servers use try/finally so the counter always decrements, even on crashes
How the Load Balancer fits in
The load balancer sits between the IDMS proxy and the 4 backend servers. Every request that passes IDMS inspection is forwarded here. The LB picks a healthy server using the selected algorithm and proxies the request. If a server is down or overloaded, it is automatically skipped — the client never sees a failure unless all 4 servers are unavailable.
