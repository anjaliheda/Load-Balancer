Even distribution
Round Robin
Cycles through servers in fixed order: 1 → 2 → 3 → 4 → 1… Simple, predictable, zero state needed. Best when all servers are equally capable and tasks take similar amounts of time.
Our improvements — click to expand each
▶
Background health checks — unhealthy servers auto-skipped
Instead of pinging every server on every request (2s timeout = potential 8s delay), a background thread checks health every 5s and caches the result. The hot path reads from cache — zero added latency.
▶
Thread-safe iterator — concurrent requests can't collide
The round-robin counter is guarded by a threading.Lock(). Without it, two simultaneous requests could both read the same counter and both go to Server 1, skipping Server 2 entirely.
▶
Exponential backoff — failed servers aren't hammered
After a health check failure the cooldown doubles: 5s → 10s → 20s → 40s… This prevents a recovering server from being overwhelmed by re-check traffic the instant it comes back.
Sticky by content
Source Hashing
Hashes the request content (task type + parameters) to deterministically pick a server. Identical requests always go to the same server — great for caching. Watch out: a single-source attacker concentrates all load on one server, saturating it before the IDMS acts.
Our improvements — click to expand each
▶
Content-aware hashing — task type + params + time bucket
Naive IP hashing sends all traffic from one client to one server. Hashing on request content (task type + parameters) provides better distribution because different task types hash to different servers, even from the same IP.
▶
Fallback on failure — tries next server in ring order
If the deterministically chosen server is unhealthy, the LB tries the next server in a ring (mod 4). The common-case determinism is preserved while avoiding hard failures.
▶
Optimised hash — first 8 hex chars of MD5
Only the first 8 hex digits of the MD5 are converted to an integer for the modulo operation. Converting all 32 chars of a full MD5 for every request was unnecessary overhead.
Adaptive by load
Least Loaded
Queries each server's active request count at routing time and picks the least busy one. Best when tasks have varying processing times (factorial takes 10× longer than addition), since round-robin would pile long tasks onto one server while others sit idle.
Our improvements — click to expand each
▶
Real-time load queries — parallel /load endpoint polling
All 4 servers are queried in parallel using concurrent.futures. Total routing overhead is ~1–3ms (the slowest server's response), not the sum of all four. Each server exposes /load returning its current active request count.
▶
Jitter tie-breaking — prevents thundering herd on "lightest" server
When servers are tied on load, a small random jitter (0–10ms) is added to each server's score before comparison. This ensures concurrent requests distribute across tied servers rather than all picking the same one and instantly creating a new hotspot.
▶
Guaranteed decrement — try/finally on every request
The server's active-request counter is incremented on arrival and decremented in a finally block. Even if the request handler raises an exception mid-execution, the counter decrements — no phantom load accumulation.
▶ Under the Hood — what we changed from the baseline implementation 4 improvements
The load balancer sits between the IDMS proxy and the 4 backend servers. Every request that passes IDMS inspection is forwarded here. If a server is down or overloaded it is automatically skipped — the client never sees a failure unless all 4 servers are unavailable simultaneously.
