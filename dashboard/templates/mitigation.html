The Mitigation Controller wraps around everything. It owns the per-request pipeline: block check → rule engine → anomaly engine → verdict execution. It short-circuits blocked IPs before wasting any detection cycles, logs every flagged event to SQLite, manages timed blocks with auto-expiry, and escalates repeat offenders.

Escalation: deprioritise → auto-block

A single anomalous request gets deprioritised (forwarded but tagged). But repeated anomalies from the same IP indicate sustained suspicious activity. The escalation system tracks "strikes" in a sliding window — when the count hits the threshold, the IP is automatically upgraded to a full timed block.

ParameterValueWhy
Strike threshold 8 strikes Buffer against jitter false positives while catching sustained bursts
Window 120 seconds Old strikes expire — a brief anomaly doesn't haunt the IP forever
Block duration 30 seconds Short for demo repeatability — production would use 5–15 minutes

Persistence & state management
SQLite Database
Two tables stored in a Docker volume (survives container restarts):
  • detection_log — every flagged event with timestamp, IP, rule, severity, action, and latency
  • ip_reputation — per-IP status (clean/flagged/blocked), block expiry, and hit count
Uses WAL mode with a persistent connection for performance. All access protected by a global lock.
In-Memory State
Fast-path data kept in memory for the dashboard:
  • Metrics ring — last 500 events (deque, O(1) append/evict)
  • Counters — outcome totals (allow/block/honeypot/deprioritise)
  • Strike windows — per-IP deque of deprioritise timestamps for escalation
  • Rate windows — per-IP request timestamps (rule engine)
  • IP baselines — per-IP IAT distributions (anomaly engine)
All cleared on Reset to support repeatable demos. Stale IP states auto-pruned every 60s.
Block duration by severity
SeverityDurationTriggered by

Blocks auto-expire. When checked and found expired, the IP is immediately cleaned up (reputation reset, rule/anomaly state cleared, strike window emptied).
