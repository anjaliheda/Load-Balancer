Rule Engine (Phase 1)

Stateless pattern matching — every request is checked against 5 rule categories in strict priority order

The Rule Engine runs first on every request, in strict priority order. It short-circuits on the first match — an unauthenticated request never reaches the injection check. If no rule fires, the request passes to the Anomaly Engine.
Detection rules — click any row to expand
P1 API Key Validation Block ►
Every request must include a valid X-API-Key header. Missing, malformed (under 8 chars, contains injection characters), or incorrect keys are rejected instantly. This is the sole authentication boundary — unauthenticated traffic never reaches any other check.
Severity: medium  •  Block duration: 60s Example trigger: X-API-Key: wrong-key-abc → 403 Forbidden
P2 SQL / NoSQL Injection Honeypot ►
Recursively extracts all strings from the JSON body and matches against 10 compiled regex patterns: UNION SELECT, DROP TABLE, OR 1=1, MongoDB operators ($where, $ne), time-based blind (SLEEP, BENCHMARK), comment markers (--, #, /*), and more. Matched requests are silently redirected to the honeypot — the attacker receives a convincing fake 200 OK while the real servers are untouched.
Severity: critical  •  Action: honeypot redirect (not block) Example trigger: num1: "1 OR 1=1 --" → 200 OK (fake response from honeypot)
P3 Rate Limiting Block ►
Sliding-window counter per IP. Timestamps are stored in a deque and evicted as they leave the window. When the count exceeds the threshold the IP is blocked. The decision is made inside the lock to prevent race conditions at the threshold boundary — two concurrent requests cannot both read 199 and both pass.
Threshold: 200 requests / 10 seconds  •  Block duration: 5 min Detection rate (experiment): 73.3% — deterministic: (750−200) / 750
P4 Endpoint Scan Detection Deprioritise ►
Detects rapid probing of multiple distinct API paths from a single IP. Attackers enumerating the API surface hit many different endpoints quickly — legitimate clients typically access 1–2 paths. A sliding window tracks distinct paths per IP. Deprioritise rather than block, since a misconfigured client might legitimately probe multiple endpoints.
Threshold: >5 distinct paths in 10s  •  Action: deprioritise + strike logged Escalation: 8 strikes in 120s → auto-block for 30s
P5 Payload Size Block ►
Two sub-checks: (a) raw body size exceeding 8 KB prevents memory exhaustion, and (b) sort_large_list requests with more than 500 items prevent CPU exhaustion. These attacks don't need to be sophisticated — repeatedly sending oversized data can degrade server performance without ever triggering the rate limiter.
Limits: 8 192 bytes body  •  500 items in list  •  Block duration: 60s
PASS All checks passed → Anomaly Engine Forward ►
If no rule fires, the request is clean from a stateless perspective. It is forwarded to the Anomaly Engine (Phase 2) for statistical scoring against this IP's baseline behaviour. The Anomaly Engine may still flag it — but that result is recorded separately.
No block. No honeypot. Rule inspect time: <1ms. Anomaly engine runs next on all requests that reach this point.
